6.0 这一章到底在讲什么

第 5 章讲了 QMSum 数据结构。你已经知道 QMSum 给的是：

turn 0
turn 1
turn 2
...

但是模型的 KV cache 不是按 turn 存的，它是按 token 位置存的。

所以第 6 章要解决的问题是：

QMSum 说答案证据在 turn 54-61，
但 KV cache 只知道 token 位置 1234-1678。
那系统怎么知道 turn 54 对应哪些 tokens？

答案就是：

turn_boundaries

它是 QMSum 的 turn 世界和 LLM 的 token/KV 世界之间的翻译表。

6.1 为什么需要 token 坐标

QMSum 原始数据的坐标是 turn 坐标。

比如一个 query 的证据标注是：

relevant_text_span = [[54, 61]]

意思是：

答案证据在 turn 54 到 turn 61。

但模型 prefill 后的 KV cache 是按 token 存的：

token 0 的 K/V
token 1 的 K/V
token 2 的 K/V
...
token N 的 K/V

KV cache 不知道什么叫 turn 54。它只知道：

第几个 token

所以工程必须建立一张桥表：

turn_idx -> token_start, token_end

这张表就是：

turn_boundaries

比如它可能长这样：

turn_boundaries[0]  = (0, 310)
turn_boundaries[1]  = (310, 318)
turn_boundaries[54] = (5200, 5348)
turn_boundaries[61] = (6100, 6180)

意思是：

turn 54 对应 prompt 里的 token [5200, 5348)
turn 61 对应 prompt 里的 token [6100, 6180)

注意这里是半开区间：

[start, end)

包含 start，不包含 end。

6.2 turn_boundaries 为什么是地基

后面很多东西都依赖 turn_boundaries。

第一，切 chunk 要靠它。

比如代码知道：

turn 54 = token [5200, 5348)

如果：

route_chunk_size = 128

那么可以切出：

chunk A = [5200, 5328)
chunk B = [5328, 5348)

第二，Q-K 打分要靠它。

因为 Q-K fine routing 不是对原始字符串打分，而是对历史 chunk 的 K 打分。

要拿某个 chunk 的 K，就必须知道：

这个 chunk 在 KV cache 里的 token start/end 是多少。

第三，评价指标也要靠它。

Q-K 选出来的是 chunk：

selected chunk = token [5200, 5328)

但 QMSum gold evidence 是 turn：

gold turns = 54-61

所以要把 selected chunk 映射回 turn，才能算：

selected_turn recall
selected_turn precision
selected_turn F1

一句话：

turn_boundaries 是 QMSum turn 世界和 LLM token/KV 世界之间的翻译表。

6.3 build_qmsum_prompt 在做什么

核心代码在 qmsum_data.py 的 build_qmsum_prompt(...)。

它先把每个 turn 拼成文本：

speaker = turn.get("speaker", "").strip() or "Speaker"
content = turn.get("content", "").strip()
turn_text = f"{speaker}: {content}\n\n"

比如：

Project Manager: Yep . Soon as I get this ...

User Interface: The prototype discussion .

Industrial Designer: I'd say a bit smaller would probably be nice .

然后把所有 turn 拼成一个大 prompt：

prompt_text = turn0_text + turn1_text + turn2_text + ...

这个 prompt_text 就是模型做 prefill 的长上下文。

同时，代码会记录每个 turn 在字符串里的字符范围：

turn_char_spans.append((char_offset, char_offset + len(turn_text)))

比如简化例子：

turn 0 chars: [0, 80)
turn 1 chars: [80, 130)
turn 2 chars: [130, 220)

这一步还是字符坐标，不是 token 坐标。

6.4 tokenizer 怎么把字符坐标变成 token 坐标

接下来代码调用 tokenizer：

encoded = tokenizer(
    prompt_text,
    add_special_tokens=False,
    return_offsets_mapping=True,
)

这里关键是：

return_offsets_mapping=True

它让 tokenizer 返回一张表：

token_idx -> 这个 token 对应 prompt_text 里的字符范围

比如玩具例子：

prompt_text = "A: hello world\n\nB: good morning"

tokenizer 可能给出：

token 0 -> chars [0, 1)    "A"
token 1 -> chars [1, 2)    ":"
token 2 -> chars [3, 8)    "hello"
token 3 -> chars [9, 14)   "world"
token 4 -> chars [16, 17)  "B"
token 5 -> chars [17, 18)  ":"
token 6 -> chars [19, 23)  "good"
token 7 -> chars [24, 31)  "morning"

如果 turn 0 的字符范围是：

turn 0 chars [0, 15)

那它覆盖 token：

token 0, 1, 2, 3

所以：

turn_boundaries[0] = (0, 4)

如果 turn 1 的字符范围是：

turn 1 chars [16, 31)

那它覆盖 token：

token 4, 5, 6, 7

所以：

turn_boundaries[1] = (4, 8)

这就是“从字符坐标到 token 坐标”的核心。

6.5 为什么 add_special_tokens=False 很重要

这个参数非常重要。

工程必须让两件事的 token 坐标一致：

1. build_qmsum_prompt 里算出来的 turn_boundaries
2. qmsum_mainline.py 里真正 prefill 的 input_ids

所以算 turn_boundaries 时使用：

add_special_tokens=False

真正 prefill 时也使用：

inputs = tokenizer(
    prompt_text,
    return_tensors="pt",
    add_special_tokens=False,
)

如果一边加 special tokens，另一边不加，就会错位。

比如 tokenizer 自动在开头加了一个 BOS token：

token 0 = <s>
token 1 = 原来的第一个文本 token
token 2 = 原来的第二个文本 token
...

那所有 turn boundary 都会整体偏 1。

这对 KV cache 是致命的，因为你以为自己拿的是 turn 54 的 K/V，实际可能偏到别的文本。

核心原则是：

算 turn_boundaries 用什么 tokenization，
prefill KV cache 也必须用同一套 tokenization。

6.6 一个完整小例子

假设有三个 turn：

turn 0:
Project Manager: Welcome everyone.

turn 1:
Designer: The remote should be yellow.

turn 2:
Marketing: Users want simple controls.

拼成 prompt：

Project Manager: Welcome everyone.

Designer: The remote should be yellow.

Marketing: Users want simple controls.

假设 tokenizer 后得到：

turn 0 -> tokens [0, 6)
turn 1 -> tokens [6, 14)
turn 2 -> tokens [14, 21)

那么：

turn_boundaries = [
  (0, 6),
  (6, 14),
  (14, 21),
]

如果 QMSum 给了一个 query：

{
  "query": "What colour should the remote be?",
  "relevant_text_span": [[1, 1]]
}

它表示：

答案证据在 turn 1。

工程就能通过：

turn_boundaries[1]

知道：

turn 1 对应 token [6, 14)

后面切 chunk、取 KV、评估 selected turns 都靠这个。

6.7 和真实工程代码怎么对应

第六章主要对应两个位置。

第一处在 qmsum_data.py：

prompt_text, turn_boundaries = build_qmsum_prompt(transcripts, tokenizer)

这里生成：

prompt_text:
  整场 meeting 拼起来的长文本。

turn_boundaries:
  每个 turn 对应的 token 范围。

第二处在 qmsum_mainline.py：

inputs = tokenizer(
    prompt_text,
    return_tensors="pt",
    add_special_tokens=False,
)
input_ids = inputs.input_ids.cuda()

这里用同一个 prompt_text 做 prefill，拿到全量 KV cache。

后续切 chunk 时，会用 turn_boundaries：

turn_idx -> start_t, end_t
然后按 route_chunk_size 切成 chunk candidates。

所以这条链是：

meeting_transcripts
-> prompt_text
-> tokenizer offset_mapping
-> turn_boundaries
-> chunk token ranges
-> KV cache slice
-> Q-K scoring
-> selected turns evaluation

6.8 本章核心记忆

第六章最核心的是：

QMSum 的证据标注是 turn 坐标；
LLM 的 KV cache 是 token 坐标；
turn_boundaries 把 turn 坐标翻译成 token 坐标。

如果没有这张表，工程就不知道：

哪个 turn 对应 KV cache 里的哪一段；
哪个 chunk 来自哪个 turn；
selected chunk 有没有覆盖 gold evidence turn。

所以：

turn_boundaries 是全工程的地基。
