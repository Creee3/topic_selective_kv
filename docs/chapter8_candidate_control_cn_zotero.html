8.0 本章一句话

第八章讲的不是 Q-K 打分公式，而是 Q-K 之前的系统准备层。

第八章 =
  在真正 Q-K 精打分之前，
  先把候选 chunk 数量压下来，
  再把 Q-K 计算准备得更快。

也就是说，本章回答两个问题：

问题 1：哪些 chunk 有资格进入 expensive Q-K scoring？
问题 2：进入 Q-K 前，怎样避免重复计算、降低 routing compute？

核心记忆：第八章是候选池控制和执行优化，不是 attention 原理本身。attention / Q-K affinity / layer-head 聚合放在第九章讲。

8.1 本章在系统里的位置

前面第七章已经做完 coarse routing。第七章输入 query，输出 selected topic。

第七章：
query
-> lexical / BM25 粗路由
-> selected_topic_ids

但是，一个 topic 里面仍然可能有很多 chunk。比如：

selected topic = topic 3

topic 3 里面可能有：
chunk 0
chunk 1
chunk 2
...
chunk 520

如果直接对这 521 个 chunk 全部做 Q-K，会很慢。第八章就是在这里接手：

topic 内候选 chunks
-> cheap lexical prefilter
-> dynamic candidate pool
-> coarse segment gate
-> Q-K execution cache / batching
-> 交给第九章 exact Q-K scoring

所以本章是第七章和第九章之间的桥。

8.3 第八章的完整漏斗

这一章最适合用漏斗来记。

全会议 chunks
  ↓
第七章已经选出的 topic
  ↓
topic filter
  ↓
lexical candidate prefilter
  ↓
dynamic candidate pool
  ↓
coarse segment gate
  ↓
query-Q cache / candidate-key cache / batching
  ↓
第九章 exact Q-K scoring

每一层都在做一件很具体的事。

步骤 作用 是否改变 Q-K 公式
topic filter 只保留 selected topic 内的 chunk 不改变
lexical prefilter 用便宜 BM25 先筛掉一批明显不相关 chunk 不改变
dynamic candidate pool 按 query 类型控制 Q-K 前候选池大小 不改变
coarse segment gate 按连续 chunk 段做候选池保护性裁剪 不改变
query-Q cache / batching 减少重复计算，提高 Q-K 执行效率 不改变

这里最容易误解的一点是：第八章很多东西会影响最终结果，因为它决定哪些 chunk 能进入 Q-K；但它并没有修改 Q-K 分数公式。

8.4 topic filter：只保留选中 topic 内的 chunk

第七章选出 topic 后，第八章第一步就是过滤 candidate。

selected_topic_ids = [3]

原始候选可能是：

chunk 0: topic 0
chunk 1: topic 1
chunk 2: topic 3
chunk 3: topic 3
chunk 4: topic 5

topic filter 后只留下：

chunk 2: topic 3
chunk 3: topic 3

代码逻辑对应：

candidates_to_score = [
    cand
    for cand in route_candidates
    if any(int(topic_id) in selected_topic_set for topic_id in cand["topic_ids"])
]

这一步没有复杂 scoring，只是先把范围限制在粗路由选中的 topic 里面。

8.5 dynamic route budget：决定最后要选多少 chunk

这一步决定第九章最终要选多少个 chunk。

summary-style query  -> effective_route_top_k = 16
detail-style query   -> effective_route_top_k = 12
balanced-style query -> effective_route_top_k = 12

注意，这里的 summary/detail/balanced 是工程内部为了预算控制做的文本启发式分类，不是 QMSum 的 general_query_list/specific_query_list。

QMSum 数据字段：
specific_query_list
general_query_list

当前主线使用：
specific_query_list

工程内部预算类型：
detail / summary / balanced

所以，一个 query 可以来自 specific_query_list，但预算类型仍然是 summary。

query:
Summarize the presentation and discussion about the design of the remote.

数据来源：
specific_query_list

预算类型：
summary

最终 chunk 预算：
effective_route_top_k = 16

另一个 detail 例子：

query:
What did Marketing think about Project Manager's system?

预算类型：
detail

最终 chunk 预算：
effective_route_top_k = 12

8.6 lexical candidate prefilter：便宜筛选 Q-K 候选

这是第八章第一层真正的候选裁剪。

route_candidate_prefilter = lexical

意思是：在 expensive Q-K 之前，先用便宜的 lexical / BM25 分数筛一批 candidate chunk。

当前固定参数是：

route_candidate_prefilter_factor = 6
route_candidate_prefilter_min_keep = 48
route_candidate_prefilter_max_keep = 128

它们都是预先设置好的工程超参数。不是模型自动学的，也不是每个 query 重新调出来的。

计算公式可以记成：

prefilter_keep =
  min(
    num_candidates,
    clamp(effective_route_top_k * factor, min_keep, max_keep)
  )

其中：

factor:
  最终 top-k 的放大倍数。
  例如最后要选 12 个 chunk，先给 Q-K 72 个候选。

min_keep:
  防止候选池太小。
  当前至少保留 48 个。

max_keep:
  防止候选池太大。
  当前最多保留 128 个。

num_candidates:
  当前 topic filter 后实际有多少候选。
  最终不能超过这个数。

一个 detail query 的具体计算：

effective_route_top_k = 12
factor = 6
min_keep = 48
max_keep = 128
num_candidates = 521

12 * 6 = 72
max(48, 72) = 72
min(128, 72) = 72
min(521, 72) = 72

所以 lexical prefilter 后保留 72 个 chunk。

这里的 72 不是最终答案 chunk，而是进入后续候选池控制和 Q-K scoring 的候选。

容易混淆：factor=6、min_keep=48、max_keep=128 是固定配置；但是最后保留多少个 chunk 是动态算出来的，因为 effective_route_top_k 和 num_candidates 会变。

8.7 dynamic candidate pool：按 query 类型继续控池

lexical prefilter 后，系统还会根据 query 类型继续控制 exact Q-K 前的候选池大小。

dynamic_candidate_pool_budget_map =
  summary:96
  detail:48
  balanced:56
  default:56

这一步的目的不是换一种 scoring，而是把 Q-K 前的候选数量控制在一个更符合 query 范围的大小。

summary query:
  问题范围大，需要覆盖更多证据。
  Q-K 前最多留 96 个候选。

detail query:
  问题范围窄，通常只找具体事实/观点。
  Q-K 前最多留 48 个候选。

balanced query:
  中间情况。
  Q-K 前最多留 56 个候选。

继续用刚才 detail 例子：

topic filter 后：521 个 chunk
lexical prefilter 后：72 个 chunk
dynamic candidate pool：
  detail target = 48

所以：
72 -> 48

如果是 summary query，可能是：

effective_route_top_k = 16
lexical prefilter:
  16 * 6 = 96

dynamic candidate pool:
  summary target = 96

所以：
96 -> 96

也就是说，dynamic candidate pool 有时会继续裁剪，有时不会裁剪。

8.8 coarse segment gate：按连续段做保护性裁剪

coarse segment gate 的核心思想是：会议证据通常是连续的，不一定应该只按单个 chunk 看。

route_coarse_segment_size = 4
route_coarse_segment_keep_ratio = 0.65
route_coarse_segment_min_keep = 64

segment_size=4 的意思是每 4 个相邻 chunk 看成一个小段：

segment 0: chunk 0, 1, 2, 3
segment 1: chunk 4, 5, 6, 7
segment 2: chunk 8, 9, 10, 11

然后系统给这些 segment 一个 lexical 分数，优先保留更可能相关的连续段。

为什么需要连续段？因为会议证据经常像这样出现：

chunk 20: 问题被提出
chunk 21: 回答开始
chunk 22: 解释原因
chunk 23: 得出结论

如果只保留单个 chunk，可能会断掉上下文。segment gate 想尽量保留连续证据。

route_coarse_segment_min_keep = 64 的单位是 chunk，不是 segment。

route_coarse_segment_min_keep = 64

意思是：
coarse segment gate 之后至少保留 64 个 candidate chunks。

不是：
至少保留 64 个 segments。

如果 segment_size=4，那么 64 个 chunk 大概相当于 16 个 segment。

64 chunks / 4 chunks per segment = 16 segments

所以如果当前候选池只有 48 个 chunk，gate 通常不会继续裁剪：

当前候选 = 48
min_keep = 64

已经不多了，所以 skip no prune。

如果当前候选是 96 个 chunk，它可能裁到 64：

96 -> 64

这一步的作用是：既想继续降低 Q-K 计算量，又不希望把连续证据切得太碎。

8.9 Q-K 执行准备：batching 与 cache

前面几步结束后，系统得到 candidates_to_score。这些候选会进入第九章 exact Q-K scoring。

但现在主线还做了三个执行优化：

qk_score_batch_size = 64
cache_candidate_keys = True
cache_query_q = True

这三个东西不改变 Q-K 分数，只改变执行方式和速度。

参数 作用 公平性判断
qk_score_batch_size=64 一次最多把 64 个 candidate chunk 放进一批 Q-K 计算 公平，只改变 GPU 执行并行度，不改变分数
cache_candidate_keys=True candidate chunk 的 K 可以缓存，避免重复准备 公平，但如果其他 Q-K baseline 需要 K，也应允许同样缓存
cache_query_q=True 同一个 query 的 Q 只算一次，后续 batch 复用 公平，因为 Q 只由 query 和模型决定，不由 candidate 决定

cache_query_q=True 是这次 TTFT 大幅下降的关键。

以前：
第 1 批 candidate：算一次 query Q
第 2 批 candidate：又算一次 query Q
第 3 批 candidate：又算一次 query Q

现在：
先算一次 query Q
第 1 批 candidate 复用
第 2 批 candidate 复用
第 3 批 candidate 复用

这个做法合理，因为同一个 query 的 Q 不取决于 candidate chunk。

score(chunk A) = Q(query) · K(chunk A)
score(chunk B) = Q(query) · K(chunk B)
score(chunk C) = Q(query) · K(chunk C)

这里的 Q(query) 是同一个。

所以 query-Q cache 是 execution-side optimization，不是新的打分公式。

8.10 一个完整数字例子

假设当前 query 是：

What did Marketing think about Project Manager's system?

它是 detail-style query。粗路由选中了：

selected topic = topic 3

topic 3 里有 521 个 candidate chunks。

1. topic filter
   topic 3 内候选数 = 521

2. dynamic route budget
   detail -> effective_route_top_k = 12

3. lexical candidate prefilter
   factor = 6
   12 * 6 = 72
   min_keep = 48
   max_keep = 128
   所以 521 -> 72

4. dynamic candidate pool
   detail target = 48
   所以 72 -> 48

5. coarse segment gate
   当前只有 48 个 chunk
   min_keep = 64
   所以通常不继续裁剪
   48 -> 48

6. Q-K execution prep
   batch_size = 64
   cache_query_q = True
   cache_candidate_keys = True

7. 进入第九章
   对 48 个 chunk 做 exact Q-K
   最后选 top 12 个 chunk

对应输出字段可以这样读：

query_budget_type = detail
effective_route_top_k = 12

num_candidates_before_prefilter = 521
num_candidates_after_prefilter = 72
num_candidates_after_dynamic_pool = 48
num_candidates_after_coarse_segment_gate = 48
num_candidates_scored_qk = 48

这个例子最能说明第八章的作用：Q-K 并没有面对 521 个 chunk，而是只面对 48 个经过筛选的候选。

8.11 固定参数、动态结果、跨数据集风险

这里要特别小心。当前这些参数是 QMSum current profile 的固定工程超参数。

factor = 6
min_keep = 48
max_keep = 128
detail pool = 48
summary pool = 96
balanced pool = 56
segment_min_keep = 64

它们不是通用真理，也不是所有数据集都一定最优。

为什么换数据集可能有问题？因为这些参数依赖当前实验形态：

QMSum 会议长度
topic 内 chunk 数量
chunk_size = 128
specific query 的证据范围
GPU 能承受的 Q-K scoring 数量
当前 TTFT / F1 trade-off 目标

举几个例子：

如果另一个数据集 topic 内只有 30 个 chunk：
  min_keep=48 基本没有意义，最后可能全保留。

如果另一个数据集 topic 内有 3000 个 chunk：
  max_keep=128 可能太小，证据召回会掉。

如果另一个数据集的问题都很宽：
  detail=48 可能不够。

如果另一个数据集的问题都很窄：
  summary=96 可能浪费。

所以论文里更稳妥的说法是：

当前主线在 QMSum profile 下使用固定的 candidate prefilter budget。
这些参数属于系统超参数，用于控制 Q-K scoring 的计算量。
跨数据集时需要重新 sweep，或者改成 latency-aware / data-adaptive budget。

未来更通用的方向可以是：

根据 topic 内 candidate 总数设比例；
根据 lexical 分数分布自动截断；
根据目标 latency budget 动态决定 pool size；
根据 query 类型和历史召回情况自适应调整。

8.12 第八章和第九章的边界

第八章和第九章必须分清楚。

章节 核心问题 关键词
第八章 哪些 chunk 能进入 Q-K？Q-K 怎么执行得更快？ candidate pool, prefilter, dynamic budget, segment gate, batching, cache
第九章 进入 Q-K 的 chunk 到底怎么打分和排名？ Q, K, attention affinity, layer aggregation, head aggregation, chunk scalar

一句话：

第八章控制候选和执行成本。
第九章计算 Q-K 分数和最终排名。

8.13 给老师汇报的版本

可以这样讲：

粗路由选中 topic 后，我不会直接对 topic 内所有 chunk 做昂贵的 Q-K。

我先用 lexical prefilter、query-type candidate budget 和 coarse segment gate 控制候选池，
把 topic 内大量 chunk 压到几十个更可能相关的候选。

然后我用 batched Q-K scoring、candidate-key cache 和 query-Q cache 优化执行。
其中 query-Q cache 是 TTFT 大幅下降的关键，因为同一个 query 的 Q 只需要计算一次，
后续所有 candidate batch 都可以复用。

这些优化不改变 Q-K 打分公式，只减少进入 Q-K 的候选数量，并消除重复计算。
因此它们主要降低 routing compute，从而降低 selected TTFT。

最后一条边界也要说清楚：

第八章是系统侧候选池控制和执行优化；
第九章才是 Q-K attention fine routing 的具体打分原理。
