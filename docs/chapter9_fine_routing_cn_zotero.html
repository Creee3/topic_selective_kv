9.0 本章一句话

第九章讲的是细路由，也就是在第八章留下来的候选 chunk 里，用模型自己的 Q-K attention affinity 做精排。

第九章 =
  对每个 candidate chunk 取历史 K，
  对当前 query 算 Q，
  用 QK^T 算 attention affinity，
  把 token / head / layer 分数聚合成 chunk 分数，
  最后选出 selected_candidates。

核心记忆：第九章不是普通 retrieval。普通 retrieval 通常在外部 embedding 空间里比较 query 和文本；你的 Q-K routing 是在 LLM 自己的 attention 空间里比较 query 的 Q 和历史 chunk 的 K。

9.1 第九章在系统里的位置

完整链条是：

第 7 章：coarse routing
  query -> 选 topic

第 8 章：candidate control
  selected topic 内 chunks -> candidates_to_score

第 9 章：fine routing
  candidates_to_score -> Q-K 精排 -> selected_candidates

所以第九章的输入不是全会议所有 chunk，而是第八章已经筛过的一批候选。

输入：
query_ids
candidates_to_score
完整历史 prompt 的 KV cache
scoring_layers
model

输出：
selected_candidates

如果第八章把 521 个 topic 内 chunk 压到 48 个，那么第九章通常只对这 48 个 chunk 做 exact Q-K。

9.2 Q-K fine routing 到底在算什么

核心公式是：

attention_affinity(query, chunk) = Q(query) K(chunk)^T / sqrt(head_dim)

直觉上就是：

query 的 Q 和某个历史 chunk 的 K 越对齐，
这个 chunk 对当前 query 越可能重要。

但是工程里不是一个简单向量点积，而是多个维度一起算：

多个 layer
多个 head
多个 query token
多个 chunk token
每个 head 内 128 维向量

当前 Mistral 类模型大概是：

hidden_size = 4096
num_attention_heads = 32
num_key_value_heads = 8
head_dim = 128

head_dim=128 来自：

4096 / 32 = 128

所以在每个 attention head 里，Q 和 K 都是 128 维向量。

9.3 exact Q-K 的代码入口

第九章最核心的入口是 qmsum_mainline_routing.py 里的：

def score_candidates_exact_qk_batched(...)

它的作用是：

对 candidates_to_score 分 batch，
取每个 candidate 的 K，
拿 query 的 Q，
计算 Q-K 分数，
返回 scored_candidates。

主线里调用它的位置大概是：

scored_candidates, per_head_scores, candidate_key_cache_stats =
    score_candidates_exact_qk_batched(
        kv_3d,
        candidates_to_score,
        query_ids,
        model,
        scoring_layers,
        split_kv,
        qk_score_fn,
        qk_aggregation=args.qk_aggregation,
        qk_topk=args.qk_topk,
        batch_size=args.qk_score_batch_size,
        candidate_key_cache=candidate_key_cache,
        cache_enabled=args.cache_candidate_keys,
        query_q_cache=query_q_cache,
        cached_qk_score_fn=cached_qk_score_fn,
    )

这里的 scored_candidates 就是每个 chunk 已经带上 Q-K scalar 分数后的候选。

9.4 candidate chunk 的 K 从哪里来

每个 candidate chunk 都有 token 坐标：

start_t
end_t
n_tokens

比如一个 candidate 可能是：

candidate_id = 728
start_t = 13624
end_t = 13666
n_tokens = 42

这表示这个 chunk 对应完整 prompt 里的 token 范围：

[13624, 13666)

系统已经对完整会议 prompt 做过 prefill，所以完整历史 KV cache 已经存在。第九章要做的是从完整 KV cache 里切出这个 chunk 的 K。

完整历史 KV cache
  -> 按 start_t / end_t 切出 candidate chunk
  -> 得到 K(chunk)

一个 chunk 的 K 大概可以理解成：

K_chunk:
  layer 0 的 K
  layer 1 的 K
  ...
  layer 31 的 K

每一层 K 的形状近似是：

(num_key_value_heads, chunk_len, head_dim)

代入当前模型就是：

(8, chunk_len, 128)

9.5 query 的 Q 从哪里来

query 先被 tokenizer 转成 token ids：

query_ids = tokenizer(query_text, return_tensors="pt").input_ids.cuda()

然后模型对 query 做一次 forward，取出对应 layer 的 hidden state，再经过这一层的 q_proj 得到 Q。

hidden = outputs.hidden_states[layer_idx]
q_tensor = q_proj(hidden)
q_tensor = q_tensor.view(1, -1, num_heads, head_dim).transpose(1, 2)

得到的 Q 形状是：

Q: (1, 32, query_len, 128)

每一维含义是：

1:
  batch size

32:
  attention heads

query_len:
  query 有多少 token

128:
  每个 head 里的向量维度

当前主线启用了 cache_query_q=True，所以同一个 query 的 Q 只算一次，然后后续 candidate batch 复用。

query-Q cache:
  对一个 query，只 forward 一次。
  一次性缓存多个 scoring layer 的 Q。
  后续所有 candidate batch 直接复用这些 Q。

9.6 QK^T 公式和代码怎么对应

核心代码是：

attn = torch.matmul(Q, K_layer.transpose(-2, -1)) / scale

其中：

scale = sqrt(head_dim)

对应公式：

attn = Q K^T / sqrt(head_dim)

如果某个 query 有 8 个 token，candidate chunk 有 42 个 token，那么形状是：

Q:
  (1, 32, 8, 128)

K:
  (1, 32, 42, 128)

K^T:
  (1, 32, 128, 42)

QK^T:
  (1, 32, 8, 42)

这个 (1, 32, 8, 42) 的意思是：

1 个 batch；
32 个 heads；
query 的每个 token；
chunk 的每个 token；
两两都有一个 affinity 分数。

所以一个 head 里会有一个 query_len × chunk_len 的矩阵。

如果 query_len = 8, chunk_len = 42：
每个 head 里有一个 8 × 42 的分数矩阵。

affinity 可以理解成未经过 softmax 的 attention 亲和度。它不是最终生成答案时真正使用的 attention 权重，而是你拿来做 routing 的相关性信号。

9.7 为什么 K 要从 8 个 head 扩到 32 个 head

当前模型不是每个 Q head 都有一个独立 KV head。它是 grouped-query attention 类结构。

num_attention_heads = 32
num_key_value_heads = 8

也就是说：

Q 有 32 个 heads；
K/V 只有 8 个 KV heads。

但是做 QK^T 的时候，Q 和 K 的 head 数需要对齐，所以代码有：

if num_kv_heads != num_heads:
    n_rep = num_heads // num_kv_heads
    K_layer = K_layer.repeat_interleave(n_rep, dim=1)

当前：

n_rep = 32 / 8 = 4

意思是：

每个 KV head 复制给 4 个 query heads 使用。

复制前：

K: (1, 8, chunk_len, 128)

复制后：

K: (1, 32, chunk_len, 128)

这样才能和 Q 的形状对齐：

Q: (1, 32, query_len, 128)
K: (1, 32, chunk_len, 128)

9.8 token 矩阵如何变成每个 head 一个分数

QK^T 得到的是四维矩阵：

attn: (1, 32, query_len, chunk_len)

但最终系统不可能为每个 chunk 保留这么大的矩阵，所以代码对 token 维度求平均：

per_head = attn.mean(dim=(2, 3)).squeeze(0)

dim=(2, 3) 分别是：

dim 2: query_len
dim 3: chunk_len

所以这一步做的是：

对所有 query token × chunk token 的 affinity 求平均。

形状变化是：

(1, 32, query_len, chunk_len)
-> (32,)

也就是：

这个 chunk 在 32 个 heads 上，各有一个分数。

注意：这里的平均是一种简化聚合方式。它把 token-level 矩阵压成 head-level 分数，方便后面按 chunk 排序。

9.9 多 layer 如何聚合

当前默认 scoring layers 是：

scoring_layers = [0, 8, 16, 24, num_model_layers - 1]

如果模型有 32 层，那么就是：

layer 0
layer 8
layer 16
layer 24
layer 31

每个 layer 都会对所有 candidates 算一遍 Q-K，得到：

(n_chunks, 32)

意思是：

每个 chunk 有 32 个 head 分数。

多层聚合代码是：

avg_scores = np.mean(all_scores, axis=0)

所以多个 layer 平均后，形状仍然是：

(n_chunks, 32)

这时的每个 head 分数已经是多个 layer 平均后的分数。

layer 0 head 0 分数
layer 8 head 0 分数
layer 16 head 0 分数
layer 24 head 0 分数
layer 31 head 0 分数

平均后：
这个 chunk 的 head 0 分数

9.10 chunk scalar 是什么

chunk scalar 就是一个 chunk 最终用于排序的单个数字分数。

前面得到的是：

一个 chunk -> 32 个 head 分数

但排序时通常需要一个数，所以要把这 32 个 head 分数聚合成一个 scalar。

代码入口是：

def aggregate_qk_scores(scores_matrix, mode="mean", topk=4):

当前默认聚合方式是：

qk_aggregation = mean

也就是：

chunk scalar = 所有 head 分数的平均值

简化例子，假设只有 4 个 head：

chunk A head scores = [0.10, 0.30, 0.20, 0.40]
chunk B head scores = [0.50, 0.10, 0.10, 0.20]

chunk A scalar = (0.10 + 0.30 + 0.20 + 0.40) / 4 = 0.25
chunk B scalar = (0.50 + 0.10 + 0.10 + 0.20) / 4 = 0.225

所以按 scalar 排序时，chunk A 排在 chunk B 前面。

代码里还支持其他聚合方式：

mode 含义
mean 所有 head 分数平均
max 取最高 head 分数
topk_mean 取最高 k 个 head 分数再平均，默认 k=4

当前主线最重要的是 mean。

9.11 route_per_head=True 在做什么

当前主线配置里：

route_per_head = True

这表示最终选择 chunk 时，不只是简单按 scalar 排序。它会让每个 head 都先提名自己认为相关的 chunk。

代码逻辑可以翻译成：

1. 对每个 head：
     找出这个 head 评分最高的 top-k chunks。

2. 把所有 head 提名出来的 chunks 做 union。

3. 再用 chunk scalar 排序。

4. 最终仍然只保留 effective_route_top_k 个 chunk。

为什么要这么做？因为不同 head 可能关注不同类型的证据。

head 0 可能更关注 speaker
head 1 可能更关注 decision
head 2 可能更关注 number
head 3 可能更关注 topic word

如果只用平均分，有些只被少数 head 强烈关注的证据可能被平均掉。route_per_head=True 给每个 head 一个参与候选提名的机会。

但它不是无限扩大传输量。最终仍然有硬上限：最多只保留 effective_route_top_k 个 chunk。

9.12 一个完整数字例子

假设第八章后剩下 4 个 candidate chunks：

chunk A
chunk B
chunk C
chunk D

为了简单，假设只有 2 个 layer、3 个 head。

Layer 0 算出来：

          head0  head1  head2
chunk A   0.8    0.2    0.1
chunk B   0.4    0.6    0.2
chunk C   0.1    0.3    0.9
chunk D   0.5    0.4    0.3

Layer 1 算出来：

          head0  head1  head2
chunk A   0.6    0.3    0.2
chunk B   0.5    0.7    0.3
chunk C   0.2    0.2    0.8
chunk D   0.4    0.5    0.4

先做 layer 平均：

chunk A:
  [(0.8+0.6)/2, (0.2+0.3)/2, (0.1+0.2)/2]
  = [0.7, 0.25, 0.15]

chunk B:
  [0.45, 0.65, 0.25]

chunk C:
  [0.15, 0.25, 0.85]

chunk D:
  [0.45, 0.45, 0.35]

再做 scalar mean：

chunk A scalar = (0.7 + 0.25 + 0.15) / 3 = 0.367
chunk B scalar = (0.45 + 0.65 + 0.25) / 3 = 0.450
chunk C scalar = (0.15 + 0.25 + 0.85) / 3 = 0.417
chunk D scalar = (0.45 + 0.45 + 0.35) / 3 = 0.417

如果最后选 top 2，单纯 scalar 排序会选：

chunk B
chunk C 或 chunk D

如果 route_per_head=True，每个 head 先提名 top 2：

head0 top2: chunk A, chunk B/D
head1 top2: chunk B, chunk D
head2 top2: chunk C, chunk D

union 后可能包含：

chunk A, chunk B, chunk C, chunk D

然后再按 scalar 排序截断 top 2：

chunk B
chunk C 或 chunk D

这个例子说明：route_per_head 的作用是给不同 head 提名机会，但最终仍然受 scalar 排序和 top-k 上限控制。

9.13 第九章最终输出是什么

第九章最终输出是：

selected_candidates

每个 selected candidate 里一般有：

candidate_id
turn_idx
local_chunk_idx
topic_ids
transfer_topic_id
transfer_node_id
start_t
end_t
n_tokens
score
prefilter_score

这些字段的作用是：

字段 含义
candidate_id 候选 chunk 的内部编号
turn_idx 这个 chunk 属于哪一个会议 turn
start_t / end_t 这个 chunk 在完整 prompt KV 里的 token 坐标
score Q-K fine routing 得到的 chunk scalar 分数
prefilter_score 第八章 lexical prefilter 的便宜分数，不是最终 Q-K 分数
transfer_topic_id / transfer_node_id 后面计算 KV 传输成本、节点数、segment 数时使用

第九章选出的这些 chunk，会进入后面的传输统计和答案生成评估。

selected_candidates
-> transfer accounting
-> selected KV fetch latency
-> selected answer context
-> selected_turn recall / answer F1 / TTFT saving

9.14 第八章和第九章的边界

第八章和第九章非常容易混在一起，所以这里再次分清楚。

章节 核心问题 主要对象
第八章 哪些 chunk 有资格进入 Q-K？Q-K 怎么执行得更快？ candidate pool, prefilter, dynamic pool, segment gate, cache, batching
第九章 进入 Q-K 的 chunk 到底怎么打分、聚合、排序？ Q, K, QK^T, affinity, layer/head aggregation, chunk scalar

一句话：

第八章决定谁能进考场；
第九章给进考场的 chunk 真正打分排名。

9.15 给老师汇报的版本

可以这样讲：

在细路由阶段，我不是用外部 embedding retrieval 给 chunk 排序，
而是复用 LLM 自己 attention 机制里的 Q-K affinity。

具体来说，query 侧经过模型对应层的 q_proj 得到 Q，
历史 chunk 侧直接使用 prefill 阶段已经缓存下来的 K。
然后对每个 candidate chunk 计算 QK^T / sqrt(d)，
得到 query tokens 和 chunk tokens 之间的 attention affinity。

这些 token-level affinity 会先在 query token 和 chunk token 维度上平均，
得到每个 head 一个分数；
再跨多个 scoring layers 平均；
最后把 head 分数聚合成一个 chunk scalar。

系统根据这些 Q-K scalar 分数选择 top chunks，
这些 chunks 对应的 KV 会被传输，用于后续 decode 或答案生成。

最后一句边界：

第九章的贡献不是重新训练模型，而是把模型内部 Q-K attention 空间作为 routing signal，
用它在 selected topic 内对 candidate chunks 做 fine-grained KV selection。
