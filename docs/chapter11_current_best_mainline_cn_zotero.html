11.0 本章一句话

当前可以记录为主线最优的一次完整实验是：QMSum train docs 0-20，每个 doc 最多 5 个 specific query，current mainline profile，4 个 virtual nodes，启用 query-Q cache 和 batched Q-K scoring。

这次实验的核心结果是：

有效 case 数: 99
selected KV: 50.89 MiB
full KV: 1662.03 MiB
KV saving: 96.53%

selected TTFT: 97.16 ms
full TTFT: 577.24 ms
TTFT saving: 80.83%

full-answer F1: 18.89%
selected-answer F1: 17.14%
answer F1 delta: -1.75%

换成人话就是：当前主线用很小的证据/KV 子集，保住了大部分回答质量，同时把 estimated TTFT 从约 577 ms 降到约 97 ms。

11.2 当前主线到底跑了什么

当前主线不是一个单独的 Q-K 打分函数，而是一整条 selective KV fetch 流程。

QMSum topic labels
-> 构造成 topic/node
-> lexical/BM25 粗路由选 top-1 topic
-> 在这个 topic 里生成 candidate chunks
-> lexical prefilter 先压候选池
-> dynamic candidate pool 继续按 query 类型控预算
-> coarse segment gate 尽量保留更集中的候选段
-> batched exact Q-K 给候选 chunks 精排
-> 选出 selected chunks
-> 按 virtual node 做 transfer accounting
-> 估算 selected KV fetch latency 和 TTFT
-> 可选：用 selected context 生成答案并算 answer F1

这条主线在代码里对应：

scripts_qmsum/run_qmsum_current_mainline.sh
  -> qmsum_mainline.py
  -> qmsum_mainline_config.py 的 mainline_profile = current
  -> qmsum_mainline_routing.py 做路由和 Q-K
  -> qmsum_eval.py 做指标和 TTFT 估算
  -> qmsum_answering.py 做回答质量评测

11.4 为什么是 99 个 case

这次实验范围是 docs 0-20，每个 doc 最多 5 个 query。

理论最大 case 数 =
  20 docs * 5 queries/doc
  = 100 cases

但实际 TSV 合并后是 99 个有效 case，因为 doc 11 只有 4 个 query case 被记录。

doc 0-10: 每个 doc 5 cases
doc 11: 4 cases
doc 12-19: 每个 doc 5 cases

total = 99 cases

所以后面所有全局平均值都是基于这 99 个有效 case 算出来的。

11.7 怎么理解“当前主线最优”

这里的“最优”要说清楚，不能说成所有可能方案里的数学全局最优。

更准确的表述是：

这是当前 mainline_profile=current 下，
最近一次完整、可复现、指标最完整的稳定主线记录。

它之所以适合作为当前主线记录，是因为：

原因 解释
范围比单 case 更可靠 不是只看一个 doc/query，而是 docs 0-20 的 99 个有效 query case
指标完整 同时有 evidence recall、answer F1、KV saving、fetch saving、TTFT saving
系统优化打开 启用了 cache_query_q、cache_candidate_keys、qk_score_batch_size=64
TTFT accounting 更合理 保留 routing_wall_clock_ms，但 system TTFT 排除 simulator-only key slicing
主线边界清楚 它明确是 simulation/prototype，不伪装成完整 serving system benchmark

汇报时不要说“我们已经找到全局最优参数”。更稳的说法是：“这是当前主线稳定配置下的最好完整记录，后续还可以继续做更大范围和更多 ablation。”

11.8 这次结果证明了什么

这次实验最强的证明点不是“回答质量超过 full context”，而是下面这个 trade-off：

极大减少 KV 传输
同时只造成较小的回答质量下降。

具体来说：

证明点 对应数字 含义
传输减少 KV saving 96.53% selected path 只需要拉极小一部分 KV
TTFT 降低 selected TTFT 97.16 ms vs full TTFT 577.24 ms routing 的额外计算被节省的 KV fetch 时间抵消，而且还有很大收益
质量没有崩 selected-answer F1 17.14% vs full-answer F1 18.89% 平均只掉约 1.75 个百分点
上下文压缩明显 context token saving 95.19% 回答阶段看到的文本上下文也大幅减少
路由时间可控 system routing 62.64 ms query-Q cache 和 batch Q-K 后，在线 routing 没有压垮 TTFT

这就是论文里可以讲的核心系统故事：

Selective KV routing trades a small amount of online routing compute
for a large reduction in remote KV fetch cost.

中文可以说：

选择性 KV 路由用一小段在线路由计算，
换来了大幅减少远端 KV 拉取量，
从而显著降低 query-time TTFT。

11.9 这次结果还不能证明什么

这一节很重要，因为老师可能会问边界。

不能证明的东西 原因
不能证明这是生产级 serving 系统 当前仍然是 simulation/prototype，没有真实远端 KV server、调度器、RDMA、GPU-to-GPU fetch
不能证明网络模型完全真实 TTFT 使用简化串行估算：bandwidth time + node RTT + segment overhead + decode startup
不能证明所有 QMSum 都稳定同样收益 这次只覆盖 train docs 0-20，后续需要更大范围 train/val/test
不能证明 evidence recall 已经足够高 selected-turn recall 平均 16.86%，说明证据定位仍是主要改进空间
不能证明 full-answer 一定比 selected-answer 好或坏 answer F1 本身受生成随机性、gold answer 风格、prompt 风格影响
不能证明 topic construction 已经自动解决 当前 topic/node 来自 QMSum labels；你的老师说这部分可以人工标注，所以当前主线把它作为给定条件

所以最稳的边界表述是：

当前结果证明的是：
在给定 topic/node 和离线 materialized KV 的设定下，
selective KV routing 可以显著降低 query-time KV fetch 和 TTFT，
同时保持接近 full-context 的回答 F1。

11.10 给老师汇报的版本

如果只汇报一小段，可以这样说：

我目前的 current mainline 是一个 QMSum 上的 selective KV routing prototype。
系统假设会议上下文已经离线 prefill，并且 KV cache 已经按 topic/chunk 形式 materialized 到若干 virtual nodes。
在线 query 到来后，系统先用 lexical/BM25 在人工或数据集给定的 topic nodes 中做粗路由，
再在选中 topic 内用 candidate prefilter、dynamic candidate pool 和 coarse segment gate 控制 Q-K 候选池，
最后用 query-Q cache + batched exact Q-K 对 chunks 精排，选择需要 fetch 的 KV。

最近一次 current mainline 完整实验覆盖 QMSum train docs 0-20，共 99 个有效 query cases。
平均 selected KV 为 50.89 MiB，而 full KV 为 1662.03 MiB，KV saving 达到 96.53%。
在 25Gbps 带宽、1ms/node RTT、0.15ms/segment overhead、15ms decode startup 的估算模型下，
selected TTFT 为 97.16 ms，full TTFT 为 577.24 ms，TTFT saving 为 80.83%。
回答质量方面，full-answer F1 为 18.89%，selected-answer F1 为 17.14%，平均只下降 1.75 个百分点。

所以这次结果说明：在当前设定下，selective KV routing 能用很小的在线 routing compute，
换来大幅减少远端 KV fetch，从而显著降低 TTFT，同时保持接近 full-context 的回答质量。
当前还不是完整 serving system，而是一个系统原型和仿真评估；后续需要扩大数据范围、补充真实分布式实现或更强网络模型。
